## 一、Python 规范（后端 + Agent）

### 核心原则

```python
# ✅ 好的代码
def get_contract_by_id(contract_id: UUID, tenant_id: UUID) -> Contract | None:
    """Retrieve a contract by its ID, scoped to the given tenant."""
    return Contract.query.filter_by(
        id=contract_id, tenant_id=tenant_id
    ).first()


# ❌ 坏的代码
def get_contract(id, tid):
    c = Contract.query.filter(Contract.id==id, Contract.tid==tid).first()
    return c
```

### 强制规则

1. 所有函数必须有 type hints
2. 所有公开函数必须有 docstring
3. 代码行长度 ≤ 120 字符
4. 使用 `black` 自动格式化
5. 使用 `ruff` 做 linting
6. 禁止 `except Exception` 裸捕获
7. LLM API 调用必须有超时处理
8. 所有外部 API 调用必须记录日志

## 二、TypeScript/React 规范（前端）

### 强制规则

1. 使用 ESLint + Prettier
2. 组件文件使用 PascalCase
3. 类型定义放在 `types/` 目录
4. 禁止 `any` 类型（全局配置 `noImplicitAny: true`）
5. API 调用统一使用 `apiClient` 封装
6. 状态管理使用 Zustand（轻量）

## 三、Git 提交规范

```
格式：<type>(<scope>): <subject>

type:
  feat     - 新功能
  fix      - Bug 修复
  docs     - 文档变更
  refactor - 代码重构
  test     - 测试变更
  chore    - 构建/配置变更

示例：
  feat(analyzer): 增加条款交叉校验逻辑
  fix(parser): 修复跨页表格解析错误
  docs(agent): 更新 Analyzer Agent Prompt 模板
```

## 四、Code Review 规范

| 检查项 | 说明 |
|---|---|
| Agent Prompt 变更 | 必须经法务顾问 Approval |
| 数据库 Schema 变更 | 必须提供 migration 脚本 |
| LLM API 调用 | 必须包含错误处理和重试逻辑 |
| 新增依赖 | 必须说明选型理由 |
| 安全相关代码 | 必须由两人以上 Review |
