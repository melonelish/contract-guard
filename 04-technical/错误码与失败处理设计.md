# 错误码与失败处理设计

> 版本：v1.0 | 最后更新：2026-06-16

---

## 一、文档目的

本文件定义 ContractGuard 的统一失败契约。

目标：

- 后端返回可预测的错误结构
- 前端知道每种错误如何展示
- Agent 在生成 API、WebSocket、Worker 代码时不再自行猜测失败语义

---

## 二、统一错误响应

```json
{
  "code": 4003,
  "message": "review failed",
  "data": null,
  "meta": {
    "request_id": "req-uuid",
    "retryable": true
  }
}
```

### 规则

- `code` 是系统内部错误码，不等于 HTTP 状态码
- `message` 是稳定英文机器语义，前端展示文案不要直接依赖它
- `meta.retryable` 表示客户端是否允许引导用户重试
- 所有 4xx/5xx 失败响应都必须带 `request_id`

---

## 三、HTTP 状态码映射

| 内部错误码 | HTTP 状态码 | 含义 | 是否可重试 |
|---|---|---|---|
| `1001 INVALID_REQUEST` | 400 | 参数不合法 | 否 |
| `1002 UNAUTHORIZED` | 401 | 未登录或登录失效 | 否 |
| `1003 FORBIDDEN` | 403 | 无权限访问资源 | 否 |
| `1004 NOT_FOUND` | 404 | 资源不存在 | 否 |
| `1005 CONFLICT` | 409 | 状态冲突/乐观锁冲突 | 是 |
| `1006 RATE_LIMITED` | 429 | 触发限流 | 是 |
| `1007 INTERNAL_ERROR` | 500 | 未分类内部错误 | 是 |
| `3004 UNSUPPORTED_FILE_TYPE` | 400 | 不支持的合同格式 | 否 |
| `3005 FILE_TOO_LARGE` | 413 | 文件超过大小限制 | 否 |
| `3006 CONTRACT_EDIT_CONFLICT` | 409 | 草稿被他人更新 | 是 |
| `4001 REVIEW_NOT_FOUND` | 404 | 审查记录不存在 | 否 |
| `4002 REVIEW_IN_PROGRESS` | 409 | 审查正在进行中 | 是 |
| `4003 REVIEW_FAILED` | 500 | 审查流程失败 | 是 |
| `4004 LLM_UNAVAILABLE` | 503 | 主备模型都不可用 | 是 |
| `5001 KB_UPDATE_FAILED` | 500 | 知识库更新失败 | 是 |

## 四、前端展示文案

| 错误码 | 用户提示 | 前端动作 |
|---|---|---|
| `1002` | 登录已失效，请重新登录 | 跳转登录页 |
| `1003` | 你没有权限查看该内容 | 停留当前页，展示无权限态 |
| `1005` | 当前操作状态已变化，请刷新后重试 | 刷新资源或重新拉取详情 |
| `1006` | 请求过于频繁，请稍后再试 | 倒计时后允许重试 |
| `3004` | 文件格式不支持，请上传 PDF、Word 或图片文件 | 阻止上传 |
| `3005` | 文件过大，请控制在 50MB 以内 | 阻止上传 |
| `3006` | 草稿已被他人修改，请刷新后再编辑 | 触发草稿刷新 |
| `4002` | 该合同正在审查中，请稍后查看结果 | 跳转进度页 |
| `4003` | 审查失败，系统已保留上下文，可稍后重试 | 展示“重试审查”按钮 |
| `4004` | AI 服务暂时不可用，请稍后再试 | 展示降级提示与重试按钮 |

## 五、WebSocket 关闭码与失败事件

### 5.1 关闭码

| 关闭码 | 含义 | 客户端动作 |
|---|---|---|
| `4001` | 票据无效或已过期 | 重新申请 WebSocket ticket |
| `4003` | 无权订阅该 `review_id` | 停止重连，提示无权限 |
| `4009` | 连接超时或服务端主动回收 | 允许自动重连一次 |
| `4500` | 服务端内部错误 | 指数退避重连 |

### 5.2 失败事件

```json
{
  "event": "failed",
  "review_id": "55555555-5555-5555-5555-555555555555",
  "task_id": "55555555-5555-5555-5555-555555555555",
  "code": 4004,
  "message": "llm unavailable",
  "detail": "主模型与备用模型均不可用",
  "retryable": true
}
```

### 5.3 规则

- WebSocket 失败时，优先发 `failed` 事件，再关闭连接
- `failed` 事件中的 `code` 使用与 HTTP 相同的内部错误码
- 前端收到 `failed` 事件后，不要继续等待 `complete`

## 六、审查流程失败矩阵

| 阶段 | 失败场景 | 返回错误码 | 状态机结果 | 恢复策略 |
|---|---|---|---|---|
| 上传 | 文件格式错误 | `3004` | 不创建审查 | 用户重新上传 |
| 上传 | 文件过大 | `3005` | 不创建审查 | 用户重新上传 |
| 审查触发 | 同合同已有进行中审查 | `4002` | 保持原审查状态 | 返回进行中的 `review_id` |
| Parser | OCR/解析超时 | `4003` | `FAILED` | 用户稍后整单重试 |
| Analyzer | 单条款超时 | 无需直接暴露 | `PARTIAL_FAIL` | 仅重试失败条款 |
| Analyzer | 全部模型不可用 | `4004` | `FAILED` | 指数退避后允许用户重试 |
| Report | 报告生成格式错误 | `4003` | `RETRYING` | 自动重试最多 2 次 |
| Validator | 校验未通过 | `4003` | `REJECTED` 后重试 | 自动回退重新生成 |
| WebSocket | 票据过期 | `4001` 关闭码 | 不影响审查主流程 | 前端重新申请票据 |

## 七、幂等与重试规则

### 7.1 幂等键

- `POST /api/v1/contracts/{id}/review`：
  幂等键 = `contract_id + latest_version_number + tenant_id`
- 相同幂等键下：
  - 若已有进行中的审查，返回原 `review_id`
  - 若最近一次审查已完成，可重新创建新 `review_id`

### 7.2 客户端重试

| 接口 | 是否允许自动重试 | 说明 |
|---|---|---|
| 上传合同 | 否 | 避免重复上传文件 |
| 触发审查 | 是 | 需带幂等键 |
| 获取审查状态 | 是 | 纯读接口 |
| 获取报告 | 是 | 纯读接口 |
| 提交反馈 | 否 | 避免重复反馈 |

## 九、实现约束

- 新增错误码必须先更新本文件，再更新 `08-impl-spec/04-api-supplement.md`
- 前端禁止直接写死英文 `message` 做业务判断，应以 `code` 为准
- WebSocket 关闭码与 HTTP 错误码不是同一命名空间，不能混用
