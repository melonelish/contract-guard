## 二、核心表 DDL

### 2.1 tenants（租户表）

```sql
CREATE TABLE tenants (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    name VARCHAR(200) NOT NULL,
    plan VARCHAR(20) NOT NULL DEFAULT 'free',
    created_at TIMESTAMP DEFAULT NOW()
);

-- RLS 策略：每个租户只看自己的数据
ALTER TABLE tenants ENABLE ROW LEVEL SECURITY;
```

### 2.2 users（用户表）

```sql
CREATE TABLE users (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    tenant_id UUID NOT NULL REFERENCES tenants(id),
    email VARCHAR(255) NOT NULL UNIQUE,
    password_hash VARCHAR(255) NOT NULL,
    name VARCHAR(100),
    role VARCHAR(20) NOT NULL DEFAULT 'member',
    created_at TIMESTAMP DEFAULT NOW()
);

CREATE INDEX idx_users_tenant ON users(tenant_id);
CREATE INDEX idx_users_email ON users(email);

-- RLS: 用户只能看本租户的用户
ALTER TABLE users ENABLE ROW LEVEL SECURITY;
CREATE POLICY tenant_isolation ON users
    FOR ALL USING (tenant_id = current_setting('app.current_tenant_id')::UUID);
```

### 2.3 contracts（合同表）

```sql
CREATE TABLE contracts (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    tenant_id UUID NOT NULL,
    user_id UUID NOT NULL REFERENCES users(id),
    title VARCHAR(500),
    file_url VARCHAR(1000) NOT NULL,
    file_type VARCHAR(10) NOT NULL,
    file_size BIGINT,
    page_count INT,
    contract_type VARCHAR(50),
    status VARCHAR(20) DEFAULT 'uploaded',
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW()
);

CREATE INDEX idx_contracts_tenant ON contracts(tenant_id);
CREATE INDEX idx_contracts_user ON contracts(user_id);

-- RLS
ALTER TABLE contracts ENABLE ROW LEVEL SECURITY;
CREATE POLICY tenant_isolation ON contracts
    FOR ALL USING (tenant_id = current_setting('app.current_tenant_id')::UUID);
```

### 2.4 reviews（审查记录表）

```sql
CREATE TABLE reviews (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    contract_id UUID NOT NULL REFERENCES contracts(id) ON DELETE CASCADE,
    status VARCHAR(20) DEFAULT 'pending',
    report_json JSONB,
    total_risks INT,
    high_risks INT,
    medium_risks INT,
    low_risks INT,
    duration_ms INT,
    started_at TIMESTAMP,
    completed_at TIMESTAMP,
    created_at TIMESTAMP DEFAULT NOW()
);

CREATE INDEX idx_reviews_contract ON reviews(contract_id);
CREATE INDEX idx_reviews_status ON reviews(status);

-- RLS: 通过 contract → tenant 间接隔离
ALTER TABLE reviews ENABLE ROW LEVEL SECURITY;
CREATE POLICY tenant_isolation ON reviews
    FOR ALL USING (
        contract_id IN (
            SELECT id FROM contracts
            WHERE tenant_id = current_setting('app.current_tenant_id')::UUID
        )
    );
```

### 2.7 contract_drafts（编辑草稿表）

> 对应 PRD S04/S05 场景：用户在编辑器中修改合同条款，需要保存草稿状态。

```sql
CREATE TABLE contract_drafts (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    contract_id UUID NOT NULL REFERENCES contracts(id) ON DELETE CASCADE,
    user_id UUID NOT NULL REFERENCES users(id),
    
    -- 草稿内容（JSON 结构，每条条款的当前编辑状态）
    -- { "clauses": [{"clause_id": "cl_001", "text": "修改后的文本", "status": "modified|accepted|pending"}] }
    draft_json JSONB NOT NULL,
    
    -- 状态
    status VARCHAR(20) DEFAULT 'editing',  -- editing | submitted | abandoned
    edit_round INT DEFAULT 1,              -- 第几轮编辑（一个合同限 3 轮）
    
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW()
);

CREATE INDEX idx_drafts_contract ON contract_drafts(contract_id);
CREATE INDEX idx_drafts_user ON contract_drafts(user_id);
CREATE UNIQUE INDEX idx_drafts_active ON contract_drafts(contract_id) WHERE status = 'editing';
    -- 一个合同同一时间只有一个活跃草稿
```

### 2.8 contract_versions（版本历史表）

> 每次提交编辑后生成新版本，支持版本对比和回溯。

```sql
CREATE TABLE contract_versions (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    contract_id UUID NOT NULL REFERENCES contracts(id) ON DELETE CASCADE,
    review_id UUID,                         -- 关联的审查记录（哪个审查触发了这次编辑）
    draft_id UUID REFERENCES contract_drafts(id),
    
    version_number INT NOT NULL,            -- v1, v2, v3...
    clauses_snapshot JSONB NOT NULL,        -- 该版本的全部条款快照
    diff_from_previous JSONB,               -- 与上一版本的差异（JSON Patch 格式）
    
    -- 差异统计
    clauses_added INT DEFAULT 0,
    clauses_removed INT DEFAULT 0,
    clauses_modified INT DEFAULT 0,
    
    created_by UUID NOT NULL REFERENCES users(id),
    created_at TIMESTAMP DEFAULT NOW()
);

CREATE INDEX idx_versions_contract ON contract_versions(contract_id);
CREATE INDEX idx_versions_number ON contract_versions(contract_id, version_number);
```

### 2.9 clause_changes（增量审查变更追踪表）

> 增量审查的核心：只对变更的条款重新调用 Analyzer Agent，未变更条款复用上次分析结果。

```sql
CREATE TABLE clause_changes (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    version_id UUID NOT NULL REFERENCES contract_versions(id) ON DELETE CASCADE,
    clause_id UUID NOT NULL REFERENCES clauses(id),
    
    change_type VARCHAR(20) NOT NULL,       -- added | removed | modified | unchanged
    old_text TEXT,                          -- 修改前的原文（modified/removed 时有值）
    new_text TEXT,                          -- 修改后的文本（added/modified 时有值）
    
    -- 增量审查结果
    need_re_review BOOLEAN DEFAULT TRUE,    -- 是否需要重新审查（modified/added=true, unchanged=false）
    re_review_result JSONB,                 -- 重新审查的结果（复用 clause_analyses 的结构）
    reused_from_review_id UUID,             -- 如果复用上次结果，记录复用来源
    
    created_at TIMESTAMP DEFAULT NOW()
);

CREATE INDEX idx_changes_version ON clause_changes(version_id);
CREATE INDEX idx_changes_need_review ON clause_changes(version_id) WHERE need_re_review = TRUE;
    -- 快速查询：本次版本有多少条款需要重新审查

### 2.11 RLS 策略总结

| 表 | RLS 隔离方式 | 策略 |
|---|---|---|
| `tenants` | 直接列 | `tenant_id = current_setting(...)` |
| `users` | 直接列 | `tenant_id = current_setting(...)` |
| `contracts` | 直接列 | `tenant_id = current_setting(...)` |
| `reviews` | 间接（通过 contracts） | 子查询 `contract_id IN (SELECT id FROM contracts WHERE tenant_id = ...)` |
| `clauses` | 间接（通过 contracts） | 同上 |
| `contract_drafts` | 间接（通过 contracts） | 同上 |
| `contract_versions` | 间接（通过 contracts） | 同上 |
| `clause_changes` | 间接（通过 contract_versions → contracts） | 二级子查询 |

### 2.12 SQLAlchemy 自动注入 tenant_id（应用层兜底）

```
┌──────────────────────────────────────┐
│         请求流                         │
│                                       │
│  HTTP Request                         │
│   │  Authorization: Bearer {jwt}      │
│   ▼                                   │
│  JWT 中间件 → 提取 tenant_id           │
│   │  request.state.tenant_id = xxx    │
│   ▼                                   │
│  DB Session 工厂                       │
│   │  SET app.current_tenant_id = xxx  │  ← 为 RLS 设置当前租户
│   ▼                                   │
│  SQLAlchemy before_execute 事件       │
│   │  自动注入 WHERE tenant_id = xxx   │  ← 应用层兜底（防止忘记加条件）│
│   ▼                                   │
│  数据库 RLS 策略                       │  ← 数据库层兜底（防止应用层绕过）│
└──────────────────────────────────────┘
```

```python
# backend/app/db/session.py
from sqlalchemy import event
from sqlalchemy.orm import Session

@event.listens_for(Session, "do_orm_execute")
def _add_tenant_filter(execute_state):
    """自动为所有查询注入 tenant_id 条件（应用层兜底）"""
    if execute_state.is_select or execute_state.is_update or execute_state.is_delete:
        # 跳过 tenants 表本身（管理后台需要看所有租户）
        for col in execute_state.statement.selected_columns:
            table = col.table
            if hasattr(table.c, "tenant_id") and not execute_state.is_superadmin:
                execute_state.statement = execute_state.statement.where(
                    table.c.tenant_id == execute_state.session.info.get("tenant_id")
                )
```

**双重保障**：应用层 SQLAlchemy 事件自动注入 + 数据库层 RLS 策略。即便应用层遗漏，数据库层截断。任何一层失效，另一层兜底。

### 2.13 增量审查流程

```
用户编辑合同 → 保存草稿 → 提交审查
      │
      ▼
┌──────────────────────────────────┐
│ 1. 版本对比                       │
│    contract_versions.diff_from_previous │
│    识别哪些条款 changed/add/remove      │
└────────────┬─────────────────────┘
             │
      ┌──────▼──────┐
      │  变更的条款   │──→ 重新调用 Analyzer Agent
      │  (need_re_review=true) │
      └─────────────┘
             │
      ┌──────▼──────┐
      │  未变的条款   │──→ 复用上次 clause_analyses 结果
      │  (reused_from_review_id) │
      └─────────────┘
             │
             ▼
      ┌──────────────┐
      │ Report Agent  │ ← 合并新分析 + 复用结果 → 生成完整报告
      └──────────────┘
```

**成本节省估算**：假设 60 条条款中用户只改了 5 条，增量审查只消耗 ~5 条款的 LLM Token（约 8% 的完整审查成本）。

## 数据库迁移方案

### Alembic 配置

```bash
# backend/alembic.ini
[alembic]
script_location = backend/migrations
sqlalchemy.url = postgresql://contractguard:changeme@localhost:5432/contractguard

# backend/migrations/env.py
# 使用 SQLAlchemy MetaData 自动检测模型变更
target_metadata = Base.metadata
```

### 迁移纪律

| 规则 | 说明 |
|---|---|
| 每次 schema 变更 = 一个新 migration | `alembic revision --autogenerate -m "add_clauses_embedding"` |
| migration 必须可逆 | 每个 `upgrade()` 必须有对应的 `downgrade()` |
| CI 自动校验 | PR 合并前运行 `alembic upgrade head`，验证无冲突 |
| 生产环境先备份 | `pg_dump` > 运行 migration > 验证 > 回滚预案 |
| 回滚演练 | 每次发布前在 staging 环境执行 upgrade → downgrade → upgrade |

### 示例迁移

```bash
# 创建迁移（本例：添加 clauses.embedding 列）
alembic revision --autogenerate -m "add_embedding_to_clauses"

# 升级到最新
alembic upgrade head

# 回滚到上一个版本
alembic downgrade -1

# 查看当前版本
alembic current
```
