## 二、技术选型决策

| 层级 | 技术 | 选型理由 |
|---|---|---|
| **前端** | React 18 + TypeScript + Ant Design | 组件生态丰富，中文支持好；微信小程序用 Taro 跨端 |
| **API Gateway** | FastAPI Middleware（MVP）→ Kong（Phase 2） | MVP 用 FastAPI 自带中间件实现限流认证；Kong 在 Phase 2 引入以统一网关管理 |
| **后端框架** | Python FastAPI | 异步高性能，AI 生态最好（LangChain/LlamaIndex均原生Python） |
| **Agent 框架** | 自研（不依赖 LangChain） | 精确控制 Agent 通信和重试逻辑，避免框架黑盒 |
| **LLM 调用** | OpenAI SDK 兼容 + 多模型适配层 | 支持 MiMo 2.5 / DeepSeek V4-Flash 热切换 |
| **消息队列** | Redis Streams | 轻量，无需额外部署 RabbitMQ/Kafka。**生产升级**：Redis Sentinel 主从 + 可选拆分为 NATS(队列)/Redis(缓存) |
| **数据库** | PostgreSQL 15 | 成熟稳定，支持 JSONB 和向量扩展 pgvector |
| **向量数据库** | Milvus（生产）/ pgvector（开发） | 支持多维过滤 + 混合检索 |
| **文件存储** | MinIO（兼容 S3） | 私有化部署友好，兼容 S3 API |
| **关键词检索** | Elasticsearch 8.x | 中文分词 + BM25 检索 |
| **部署** | Docker Compose（开发）→ K8s（生产） | 渐进式架构 |
| **监控** | Prometheus + Grafana + LangSmith | 全链路可观测 |

## 三、已知 Gap 与版本路线

| Gap | 当前方案 | 计划 |
|---|---|---|
| API Gateway | FastAPI 中间件（CORSMiddleware + 自定义 RateLimitMiddleware） | Phase 2 引入 Kong（认证/限流/路由/日志插件化） |

## 四、安全架构

### 4.1 数据安全环

```
 网络层      TLS 1.3 全程加密
     │
 应用层      JWT 认证 + RBAC 权限
     │
 数据层      AES-256-at-rest 加密存储
     │        合同原文加密、脱敏数据单独存储
     │
 审计层      全操作日志、不可删除、不可篡改
```

### 4.2 多租户隔离

```
           ┌────────────┐
           │   Shared    │ ← 法律法规库（所有租户共享，只读）
           │   Layer     │
           └─────┬──────┘
                 │
     ┌───────────┼───────────┐
     │           │           │
┌────▼────┐ ┌────▼────┐ ┌────▼────┐
│ Tenant  │ │ Tenant  │ │ Tenant  │
│    A    │ │    B    │ │    C    │
│         │ │         │ │         │
│ 合同库   │ │ 合同库   │ │ 合同库   │
│ 审查记录 │ │ 审查记录 │ │ 审查记录 │
│ 自定义   │ │ 自定义   │ │ 自定义   │
│ 规则     │ │ 规则     │ │ 规则     │
└─────────┘ └─────────┘ └─────────┘
```

SQL 查询自动注入 `WHERE tenant_id = {current_tenant}`，数据库行级安全策略兜底。

## 5. Worker Auto-Scaling Strategy

### 5.1 扩缩容决策矩阵

| Agent | 基础实例 | 最大实例 | 扩容触发 | 缩容触发 |
|---|---|---|---|---|
| Parser | 2 | 10 | 队列深度 > 5 | 队列深度 = 0 持续 5 分钟 |
| Analyzer | 5 | 30 | 队列深度 > 10 | 队列深度 < 3 持续 5 分钟 |
| Report | 2 | 5 | 队列深度 > 3 | 队列深度 = 0 持续 10 分钟 |
| Validator | 1 | 3 | 队列深度 > 2 | 队列深度 = 0 持续 10 分钟 |
| Drafter | 1 | 5 | 队列深度 > 2 | 队列深度 = 0 持续 10 分钟 |

### 5.2 扩容约束

```
全局约束:
  - 总 Analyzer 实例不超过 30（防止 LLM API 超额）
  - 总 LLM Token 消耗速率不超过 100,000 TPM
  - 扩容冷却时间: 60s（防止抖动）
  - 缩容冷却时间: 300s（防止反复创建销毁）

K8s HPA 伪配置:
  minReplicas: 2
  maxReplicas: 10
  metrics:
    - type: Pods
      pods:
        metric: queue_depth
        target: 5
```

### 5.3 冷启动优化

```
每个 Worker 实例启动时:
1. 预加载 Agent System Prompt（从 Redis 缓存）
2. 预连接知识库（Milvus + ES 连接池预热）
3. 30s 内进入 Ready 状态 → 开始消费队列
```

## 6. 日志与可观测性

### 6.1 日志层级

| 层 | 工具 | 输出 | 用途 |
|---|---|---|---|
| 应用日志 | `structlog` (结构化 JSON) | stdout → Promtail → Loki | Agent 调用链、LLM Token 消耗、错误堆栈 |
| 审计日志 | PostgreSQL `audit_logs` 表 | 数据库（不可删改） | 操作审计（登录/上传/审查/导出） |
| 基础设施日志 | Docker logs → Promtail | Loki | 容器状态、OOM、重启 |
| 指标 | Prometheus `/metrics` | Grafana 仪表盘 | API 延迟、LLM 成功率、队列深度 |

### 6.2 结构化日志格式

```python
# backend/app/logging.py
import structlog

logger = structlog.get_logger()

# Agent 调用
logger.info("agent.call",
    agent="analyzer",
    task_id="task-abc123",
    model="mimo2.5",
    prompt_tokens=1200,
    completion_tokens=800,
    latency_ms=3500,
    cost=0.024)

# 错误
logger.error("agent.failure",
    agent="analyzer",
    task_id="task-abc123",
    error_type="TimeoutError",
    retry_count=2)
```

### 6.3 日志级别规范

| 级别 | 使用场景 |
|---|---|
| `DEBUG` | Agent 内部推理步骤、RAG 检索结果 |
| `INFO` | Agent 调用起止、LLM Token 消耗、审查完成 |
| `WARNING` | 重试、降级、合同质量低（OCR<0.7） |
| `ERROR` | Agent 失败（3次重试用尽）、LLM 不可用、DB 连接失败 |
| `CRITICAL` | Vault 不可用、所有 LLM 模型不可用 |
