## 二、技术选型决策

| 层级 | 技术 | 选型理由 |
|---|---|---|
| **前端** | React 18 + TypeScript + Ant Design | 组件生态丰富，中文支持好；微信小程序用 Taro 跨端 |
| **API Gateway** | FastAPI Middleware（MVP）→ Kong（Phase 2） | MVP 用 FastAPI 自带中间件实现限流认证；Kong 在 Phase 2 引入以统一网关管理 |
| **后端框架** | Python FastAPI | 异步高性能，AI 生态最好（LangChain/LlamaIndex均原生Python） |
| **Agent 框架** | 自研（不依赖 LangChain） | 精确控制 Agent 通信和重试逻辑，避免框架黑盒 |
| **LLM 调用** | OpenAI SDK 兼容 + 多模型适配层 | 支持 MiMo 2.5 / DeepSeek V4-Flash 热切换 |
| **消息队列** | Redis Streams | 轻量，无需额外部署 RabbitMQ/Kafka。**生产升级**：Redis Sentinel 主从 + 可选拆分为 NATS(队列)/Redis(缓存) |
| **数据库** | PostgreSQL 15 | 成熟稳定，支持 JSONB 和向量扩展 pgvector |
| **向量数据库** | Milvus（生产）/ pgvector（开发） | 支持多维过滤 + 混合检索 |
| **文件存储** | MinIO（兼容 S3） | 私有化部署友好，兼容 S3 API |
| **关键词检索** | Elasticsearch 8.x | 中文分词 + BM25 检索 |
| **部署** | Docker Compose（开发）→ K8s（生产） | 渐进式架构 |
| **监控** | Prometheus + Grafana + LangSmith | 全链路可观测 |

## 五、Worker Auto-Scaling Strategy

### 5.1 扩缩容决策矩阵

| Agent | 基础实例 | 最大实例 | 扩容触发 | 缩容触发 |
|---|---|---|---|---|
| Parser | 2 | 10 | 队列深度 > 5 | 队列深度 = 0 持续 5 分钟 |
| Analyzer | 5 | 30 | 队列深度 > 10 | 队列深度 < 3 持续 5 分钟 |
| Report | 2 | 5 | 队列深度 > 3 | 队列深度 = 0 持续 10 分钟 |
| Validator | 1 | 3 | 队列深度 > 2 | 队列深度 = 0 持续 10 分钟 |
| Drafter | 1 | 5 | 队列深度 > 2 | 队列深度 = 0 持续 10 分钟 |

### 5.2 扩容约束

```
全局约束:
  - 总 Analyzer 实例不超过 30（防止 LLM API 超额）
  - 总 LLM Token 消耗速率不超过 100,000 TPM
  - 扩容冷却时间: 60s（防止抖动）
  - 缩容冷却时间: 300s（防止反复创建销毁）

K8s HPA 伪配置:
  minReplicas: 2
  maxReplicas: 10
  metrics:
    - type: Pods
      pods:
        metric: queue_depth
        target: 5
```

### 5.3 冷启动优化

```
每个 Worker 实例启动时:
1. 预加载 Agent System Prompt（从 Redis 缓存）
2. 预连接知识库（Milvus + ES 连接池预热）
3. 30s 内进入 Ready 状态 → 开始消费队列
```

### 6.1 日志层级

| 层 | 工具 | 输出 | 用途 |
|---|---|---|---|
| 应用日志 | `structlog` (结构化 JSON) | stdout → Promtail → Loki | Agent 调用链、LLM Token 消耗、错误堆栈 |
| 审计日志 | PostgreSQL `audit_logs` 表 | 数据库（不可删改） | 操作审计（登录/上传/审查/导出） |
| 基础设施日志 | Docker logs → Promtail | Loki | 容器状态、OOM、重启 |
| 指标 | Prometheus `/metrics` | Grafana 仪表盘 | API 延迟、LLM 成功率、队列深度 |

### 6.3 日志级别规范

| 级别 | 使用场景 |
|---|---|
| `DEBUG` | Agent 内部推理步骤、RAG 检索结果 |
| `INFO` | Agent 调用起止、LLM Token 消耗、审查完成 |
| `WARNING` | 重试、降级、合同质量低（OCR<0.7） |

## 七、Prompt 缓存层

### 7.1 问题

同一份合同的 Parser 输出（通常 30-50KB 结构化 JSON）会被 Analyzer、Report、Validator 反复传入。4 次冗余传输 → 每次多烧 ~15K tokens → 一份 15 页合同多花 ~¥0.06 API 费（看似不多，乘上每日 100 份审查就是 ¥6/天浪费）。

```
当前：每次 Agent 调用都传入完整 Parser JSON
  Analyzer(1)  ← 完整 JSON（50KB）
  Analyzer(2)  ← 完整 JSON（50KB）  ← 重复！
  ...
  Report       ← 完整 JSON（50KB）  ← 重复！
  Validator    ← 完整 JSON（50KB）  ← 重复！
```

### 7.2 方案：引用了代理 + 按需展开

```
Parser 输出 → Redis 缓存（key = contract:{contract_id}:parsed, TTL=1h）
                    │
    ┌───────────────┼───────────────┐
    ▼               ▼               ▼
 Analyzer        Report         Validator
 只传引用ID      只传引用ID      只传引用ID
 + 需要的字段    + 需要的字段    + 需要的字段
 子集           子集           子集
```

### 7.3 Wrapper 层实现

```python
# backend/app/llm/prompt_cache.py
import hashlib, json
from redis import Redis

redis = Redis.from_url(settings.REDIS_URL)

class PromptCache:
    """将 Parser 输出缓存，Agent 调 LLM 时只传引用 + 需要的字段"""
    
    CACHE_TTL = 3600  # 1 小时（合同审查的生命周期内有效）
    
    @staticmethod
    def store(contract_id: str, parsed_json: dict) -> str:
        """缓存 Parser 输出，返回引用 ID"""
        fingerprint = hashlib.sha256(
            json.dumps(parsed_json, sort_keys=True).encode()
        ).hexdigest()[:16]
        key = f"prompt:contract:{contract_id}:{fingerprint}"
        redis.setex(key, PromptCache.CACHE_TTL, json.dumps(parsed_json))
        return key
    
    @staticmethod
    def build_prompt(contract_id: str, agent: str, 
                     extra_context: dict = None) -> dict:
        """Agent 调用时：只从缓存取需要的字段，减少 token 消耗"""
        reference = f"prompt:contract:{contract_id}"
        cached = redis.get(reference)
        
        if not cached:
            raise CacheMissError("Parser output expired, re-run Parser")
        
        parsed = json.loads(cached)
        
        # 按 Agent 类型只取需要的字段子集
        FIELD_MAP = {
            "analyzer": ["clauses", "meta", "parties"],
            "report":    ["clauses", "meta"],           # 不需要 parties
            "validator": ["clauses"],                    # 只要条款
        }
        
        subset = {k: parsed[k] for k in FIELD_MAP.get(agent, parsed.keys())}
        return {"contract_ref": reference, "data": subset}
```

### 7.4 Token 节省估算

| Agent | 无缓存传入 | 缓存后传入 | 节省 |
|---|---|---|---|
| Analyzer ×60 条款 | ~20K tokens/次 | ~8K tokens/次（只传条款+元数据） | 60% |
| Report | ~25K tokens | ~10K tokens（去重后） | 60% |
| Validator | ~25K tokens | ~6K tokens（只要条款） | 76% |
| **单次审查总计** | **~80K tokens** | **~24K tokens** | **70%** |

### 7.5 边界处理

- **缓存击穿**：Parser 变更（用户编辑后重审）→ 旧缓存自动失效，生成新引用
- **并发安全**：Redis SET NX 防竞态，多 Worker 同时解析同一合同时只有第一个写入缓存
- **内存上限**：Redis `maxmemory-policy volatile-ttl`，Prompt 缓存设置 TTL 自动过期
