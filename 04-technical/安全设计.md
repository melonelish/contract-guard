## 2. Authentication & Authorization

### 2.1 JWT Authentication Flow

```
User Login → Verify email/password → Generate JWT (valid 24h)
                              │
                              ├── Access Token: 24h
                              └── Refresh Token: 7d

API Request: Authorization: Bearer {access_token}
Token Expired → Use Refresh Token to exchange for new Access Token
```

### 2.2 RBAC Permission Model

| Role | Permission Scope |
|---|---|
| `admin` | Manage tenants, view all data, configure rules |
| `manager` | Manage team members, view team review statistics |
| `member` | Upload contracts, view own review reports |
| `viewer` | Read-only access to shared reports |

## 3. Data Security

### 3.1 Contract File Encryption

```python
# Encrypt before storage
encrypted = AES256_GCM.encrypt(
    plaintext=file_content,
    key=tenant_encryption_key,
    nonce=os.urandom(12)
)

# Decrypt on read (authorized users only)
plaintext = AES256_GCM.decrypt(
    ciphertext=encrypted,
    key=tenant_encryption_key,
    nonce=nonce
)
```

### 3.2 Sensitive Information Masking

Auto-detect and mask before storage:

| Data Type | Detection Regex | Masking Method |
|---|---|---|
| ID Card Number | `\d{17}[\dXx]` | Keep first 3, last 4 |
| Phone Number | `1[3-9]\d{9}` | Keep first 3, last 4 |
| Bank Card Number | `\d{16,19}` | Keep last 4 digits |
| Unified Social Credit Code | `[0-9A-HJ-NPQRTUWXY]{18}` | Public info, do not mask |

## 4. LLM Security

### 4.1 Prompt Injection Protection

```
Input Layer Protection:
  · Contract files scanned for malicious text on upload
  · User-defined rules undergo HTML escaping
  · All user input prompts limited to 500 characters

Output Layer Protection:
  · Validator Agent performs content review on all outputs
  · Detect and remove any Prompt injection residue
  · Strictly forbid LLM output containing system prompt text
```

### 4.2 Model Output Filtering

| Risk | Filtering Method |
|---|---|
| Unauthorized legal opinions | Detect conclusive language like "should sue" or "constitutes a crime"; replace with "recommend consulting a lawyer" |
| User data leakage | Output must not contain other users' contract content |
| Harmful content | General content safety API filtering |

## 7. WebSocket Authentication

### 7.1 Problem

The API design includes a WebSocket endpoint for real-time review progress push (`ws://host/ws/review/{task_id}`). Standard JWT Bearer tokens in HTTP headers are incompatible with the WebSocket handshake (no custom headers in browser WebSocket API).

### 7.2 Solution: Token-in-Query + One-Time Ticket

```
Step 1: Client requests WebSocket ticket via REST
  POST /api/ws/ticket
  Authorization: Bearer {jwt}
  → Returns { ticket: "ws_tkt_xxx", expires_in: 60 }

Step 2: Client opens WebSocket with ticket
  ws://host/ws/review/{task_id}?ticket=ws_tkt_xxx

Step 3: Server validates ticket (one-time use, 60s TTL)
  · Ticket issued per session, bound to user_id + task_id
  · Used once → invalidated immediately
  · Expired tickets rejected with 4001 close code
```

### 7.3 Server-Side Implementation (Pseudocode)

```python
# FastAPI WebSocket endpoint with ticket auth
@router.websocket("/ws/review/{task_id}")
async def review_progress(ws: WebSocket, task_id: str, ticket: str = Query(...)):
    # Validate ticket
    payload = await validate_ws_ticket(ticket, task_id)
    if not payload:
        await ws.close(code=4001, reason="Invalid or expired ticket")
        return
    
    user_id = payload["user_id"]
    await ws.accept()
    
    # Subscribe to Redis pub/sub for this task
    pubsub = redis.pubsub()
    await pubsub.subscribe(f"review:{task_id}")
    
    async for message in pubsub.listen():
        await ws.send_json(message["data"])
```

### 7.4 Security Properties

| Property | Mechanism |
|---|---|
| **Authentication** | One-time ticket derived from JWT session |
| **Authorization** | Ticket bound to `user_id` + `task_id`; user cannot subscribe to others' tasks |
| **Replay protection** | Ticket single-use with Redis SET NX |
| **Expiry** | Ticket TTL 60 seconds; connection max lifetime 30 minutes |
| **Rate limiting** | Max 3 concurrent WebSocket connections per user |
