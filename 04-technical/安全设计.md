## 二、认证与授权

### 2.1 JWT 认证流程

```
用户登录 → 验证邮箱/密码 → 生成 JWT (有效期 24h)
                              │
                              ├── Access Token: 24h
                              └── Refresh Token: 7d

API 请求：Authorization: Bearer {access_token}
Token 过期 → 使用 Refresh Token 换新 Access Token
```

> **⚠️ 开发/生产分离**：开发环境使用 HS256（对称签名），生产环境必须切换为 RS256（非对称签名：私钥签名，公钥验证）。HS256 的签名密钥和验证密钥相同，一旦 JWT_SECRET 泄露攻击者可伪造任意 Token。RS256 公钥可公开分发，私钥由 Vault 管理。

### 2.2 CSRF 防护

> 本系统 API 使用 `Authorization: Bearer {jwt}` 方式认证，不依赖 Cookie 传递令牌，浏览器不会自动附加 Authorization header 到跨域请求中，因此**不存在 CSRF 攻击面**。无需额外 CSRF Token 机制。
> 
> 唯一例外：WebSocket 握手升级时若浏览器携带 Cookie 用于认证（非当前方案），则需要 CSRF 校验。当前设计使用 Token-in-Query 票据方案（见第七章），Cookie 不参与鉴权。

### 2.3 RBAC 权限模型

| 角色 | 权限范围 |
|---|---|
| `admin` | 管理租户、查看所有数据、配置规则 |
| `manager` | 管理本团队成员、查看团队审查统计 |
| `member` | 上传合同、查看自己的审查报告 |
| `viewer` | 只读查看被分享的报告 |

## 三、数据安全

### 3.1 合同文件加密

```python
# 存储前加密
encrypted = AES256_GCM.encrypt(
    plaintext=file_content,
    key=tenant_encryption_key,
    nonce=os.urandom(12)
)

# 读取时解密（仅授权用户）
plaintext = AES256_GCM.decrypt(
    ciphertext=encrypted,
    key=tenant_encryption_key,
    nonce=nonce
)
```

### 3.2 敏感信息脱敏

在存储前自动检测并脱敏：

| 数据类型 | 检测正则 | 脱敏方式 |
|---|---|---|
| 身份证号 | `\d{17}[\dXx]` | 保留前3后4 |
| 手机号 | `1[3-9]\d{9}` | 保留前3后4 |
| 银行卡号 | `\d{16,19}` | 保留后4位 |
| 统一信用代码 | `[0-9A-HJ-NPQRTUWXY]{18}` | 公开信息，不脱敏 |

## 四、LLM 安全

### 4.1 Prompt 注入防护

```
输入层防护：
  · 合同文件在上传时扫描恶意文本
  · 用户自定义规则需要经过 HTML 转义
  · 所有用户输入的提示词长度限制 500 字符

输出层防护：
  · Validator Agent 对所有输出进行内容审查
  · 检测并移除可能包含的 Prompt 注入残留
  · 严禁 LLM 输出中包含系统提示词原文
```

### 4.2 模型输出过滤

| 风险 | 过滤方式 |
|---|---|
| 越权法律意见 | 检测"应起诉""已构成犯罪"等结论性表述，替换为"建议咨询律师" |
| 用户数据泄露 | 输出中不应包含其他用户的合同内容 |
| 有害内容 | 通用内容安全 API 过滤 |

## 七、WebSocket 认证

### 7.1 问题

接口设计中包含了 WebSocket 端点用于实时推送审查进度（`ws://host/ws/review/{review_id}`），但浏览器的 WebSocket API 不支持自定义 HTTP headers，标准 JWT Bearer 令牌无法在握手阶段传递。

### 7.2 方案：Token-in-Query + 一次性票据

```
Step 1: 客户端通过 REST 请求 WebSocket 票据
  POST /api/ws/ticket
  Authorization: Bearer {jwt}
  → 返回 { ticket: "ws_tkt_xxx", expires_in: 60 }

Step 2: 客户端带票据打开 WebSocket
  ws://host/ws/review/{review_id}?ticket=ws_tkt_xxx

Step 3: 服务端校验票据（一次性使用，60 秒 TTL）
  · 票据按会话签发，绑定 user_id + review_id + task_id
  · 使用一次后立即作废
  · 过期票据返回 4001 关闭码
```

### 7.3 服务端实现（伪代码）

```python
@router.websocket("/ws/review/{review_id}")
async def review_progress(ws: WebSocket, review_id: str, ticket: str = Query(...)):
    # 验证票据
    payload = await validate_ws_ticket(ticket, review_id)
    if not payload:
        await ws.close(code=4001, reason="票据无效或已过期")
        return
    
    user_id = payload["user_id"]
    await ws.accept()
    
    # 订阅 Redis pub/sub 频道
    pubsub = redis.pubsub()
    await pubsub.subscribe(f"review:{task_id}")
    
    async for message in pubsub.listen():
        await ws.send_json(message["data"])
```

### 7.4 安全属性

| 属性 | 机制 |
|---|---|
| **身份认证** | 基于 JWT 会话签发一次性票据 |
| **权限控制** | 票据绑定 user_id + review_id + task_id，用户无法订阅他人任务 |
| **重放防护** | 票据一次性使用，Redis SET NX 保证 |
| **过期策略** | 票据 60 秒 TTL；连接最长 30 分钟 |
| **限流** | 每用户最多 3 个并发 WebSocket 连接 |
