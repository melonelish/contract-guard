## 二、为什么不用 LangChain？

| 维度 | LangChain | 自研方案 |
|---|---|---|
| 学习成本 | 高（抽象层太多） | 低（直观的 Agent 类） |
| 调试难度 | 黑盒，出错难定位 | 全链路可控 |
| 性能开销 | 链式调用 + 序列化开销大 | 直接调用，无中间层 |
| 定制灵活性 | 受框架约束 | 完全自由 |
| 版本稳定性 | 频繁 Breaking Changes | 自己控制 |
| **适用场景** | 快速原型验证 | **生产级企业应用 ← 我们** |

## 三、为什么用 Redis Streams 而不是 RabbitMQ？

| 维度 | Redis Streams | RabbitMQ |
|---|---|---|
| 部署复杂度 | 已有 Redis，无需额外部署 | 需额外部署 RabbitMQ |
| 性能 | 百万级消息/秒 | 万级消息/秒 |
| 持久化 | RDB + AOF | 内置持久化 |
| 消息确认 | XACK 机制 | ACK 机制 |
| 适用场景 | **轻量级、初创团队** | 大型分布式系统 |

## 四、LLM 模型选择

### 4.1 主模型：小米 MiMo 2.5

| 维度 | 评估 |
|---|---|
| 中文法律理解 | 强（合同条款、法条术语理解准确） |
| JSON 输出稳定性 | 高（结构化输出能力好） |
| 推理深度 | 满足合同条款分析需求 |
| 成本 | 中等（对比 GPT-4o 便宜约 60%） |
| 上下文窗口 | 128K tokens（覆盖绝大多数合同） |
| 适用任务 | 条款分析、交叉校验、报告校验 |

### 4.2 备选模型：DeepSeek V4-Flash

| 维度 | 评估 |
|---|---|
| 中文能力 | 强，原生中文模型 |
| 推理速度 | 快（Flash 版主打低延迟） |
| 成本 | 极低（约为 MiMo 2.5 的 1/5） |
| 适用任务 | 文档解析、报告生成、文本处理 |

### 4.3 任务分配

| 任务 | 模型 | 理由 |
|---|---|---|
| 文档解析（非表格） | DeepSeek V4-Flash | 成本极低，文本处理够用 |
| 文档解析（含复杂表格） | MiMo 2.5 | 表格理解能力强 |
| 法律条款分析 | MiMo 2.5 | 推理要求高，中文法律理解准 |
| 交叉校验 | MiMo 2.5 | 需要法律逻辑推理 |
| 报告生成 | DeepSeek V4-Flash | 结构化文本生成，成本低 |
| 报告校验 | MiMo 2.5 | 质量检查不容出错 |

### 4.4 API Key 管理

主备两个 Key 通过 HashiCorp Vault 管理，代码零硬编码：

```python
# Vault 路径：secret/llm/prod/
{
  "mimo2.5": {"api_key": "来自 Vault", "base_url": "https://api.xiaomi.com/v1"},
  "deepseek-v4-flash": {"api_key": "来自 Vault", "base_url": "https://api.deepseek.com/v1"}
}
```

详见《04-technical/LLM Unified Wrapper Layer Design.md》和《01-business/收费模式与成本风控.md》。

## 五、数据库选型说明

### PostgreSQL 为什么是唯一选择？

| 需求 | PostgreSQL 能力 |
|---|---|
| 结构化数据 | 用户、合同、审查记录、团队 |
| 半结构化数据 | JSONB 存储审查报告、Agent调用日志 |
| 向量检索 | pgvector 扩展（开发环境替代 Milvus） |
| 全文检索 | 内置 tsvector（简单搜索场景） |
| 行级安全 | RLS 实现多租户数据隔离 |
| 扩展性 | 支持读写分离、分片 |

## 缓存策略

### 缓存分层

| 层级 | 存储 | TTL | 缓存内容 |
|---|---|---|---|
| L1 — 应用缓存 | Redis | 60s | 法条原文（查过一次后缓存） |
| L2 — API Key 缓存 | Vault + 内存 | 60s | LLM API Key（从 Vault 读取后缓存，支持热轮换） |
| L3 — 模板缓存 | Redis | 1h | 审查报告模板、固定文本（免责声明等） |
| L4 — 合同模板缓存 | Redis | 24h | 用户自定义合同模板 |
| L5 — 会话缓存 | Redis | 24h | 用户 JWT 令牌、WebSocket ticket |

### 缓存淘汰策略

```
Redis maxmemory-policy: allkeys-lru
最大内存: 512MB (开发) / 2GB (生产)

法条原文缓存命中率目标: >80%
LLM 响应不予缓存（每次审查必须重新推理）
```

### 缓存预热

```
服务启动时:
1. 预加载所有现行法律法规目录 → Redis Set
2. 预加载 Top 100 最常用法条原文 → Redis String
3. 健康检查验证缓存可用性
```
