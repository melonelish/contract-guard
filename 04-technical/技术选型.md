## 2. Why Not LangChain?

| Dimension | LangChain | Self-built Solution |
|---|---|---|
| Learning Curve | High (too many abstraction layers) | Low (intuitive Agent classes) |
| Debugging Difficulty | Black box, hard to pinpoint errors | Full-chain controllable |
| Performance Overhead | Chain invocation + serialization overhead | Direct invocation, no middleware |
| Customization Flexibility | Constrained by framework | Completely free |
| Version Stability | Frequent Breaking Changes | Self-controlled |
| **Best Use Case** | Rapid prototyping | **Production-grade enterprise apps ← Us** |

---

## 3. Why Redis Streams Instead of RabbitMQ?

| Dimension | Redis Streams | RabbitMQ |
|---|---|---|
| Deployment Complexity | Already have Redis, no extra deployment | Requires separate RabbitMQ deployment |
| Performance | Millions of messages/sec | Tens of thousands/sec |
| Persistence | RDB + AOF | Built-in persistence |
| Message Acknowledgment | XACK mechanism | ACK mechanism |
| **Best Use Case** | **Lightweight, startup teams** | Large-scale distributed systems |

## 4. LLM Model Selection

### 4.1 Primary Model: Xiaomi MiMo 2.5

| Dimension | Assessment |
|---|---|
| Chinese Legal Comprehension | Strong (accurate understanding of contract clauses and legal terminology) |
| JSON Output Stability | High (good structured output capability) |
| Reasoning Depth | Meets contract clause analysis requirements |
| Cost | Moderate (~60% cheaper than GPT-4o) |
| Context Window | 128K tokens (covers the vast majority of contracts) |
| Applicable Tasks | Clause analysis, cross-validation, report validation |

### 4.2 Backup Model: DeepSeek V4-Flash

| Dimension | Assessment |
|---|---|
| Chinese Ability | Strong, native Chinese model |
| Inference Speed | Fast (Flash variant optimized for low latency) |
| Cost | Very low (~1/5 of MiMo 2.5) |
| Applicable Tasks | Document parsing, report generation, text processing |

### 4.3 Task Allocation

| Task | Model | Rationale |
|---|---|---|
| Document parsing (non-table) | DeepSeek V4-Flash | Minimal cost; sufficient for text processing |
| Document parsing (complex tables) | MiMo 2.5 | Strong table comprehension |
| Legal clause analysis | MiMo 2.5 | High reasoning requirement, accurate Chinese legal understanding |
| Cross-validation | MiMo 2.5 | Requires legal logic reasoning |
| Report generation | DeepSeek V4-Flash | Structured text generation, low cost |
| Report validation | MiMo 2.5 | Quality checks cannot afford errors |

## 5. Database Selection Notes

### Why PostgreSQL Is the Only Choice?

| Requirement | PostgreSQL Capability |
|---|---|
| Structured Data | Users, contracts, review records, teams |
| Semi-structured Data | JSONB for storing review reports, Agent call logs |
| Vector Search | pgvector extension (dev environment Milvus alternative) |
| Full-text Search | Built-in tsvector (simple search scenarios) |
| Row-Level Security | RLS for multi-tenant data isolation |
| Scalability | Read/write separation, sharding |
