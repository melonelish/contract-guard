## 一、核心定价理念

**不是订阅，是按次买审查。用户花 19 块钱获得的不是一次 API 调用，而是一份合同的完整审查闭环（3 轮审-改-再审）。**

### 为什么按次而不是订阅

| | 月订阅 ¥99 | 按次单买 ¥19/次 |
|---|---|---|
| 一年只用 2 次的用户 | ¥1,188（亏炸）→ 流失 | ¥38（合理）→ 留存 |
| 一年用 50 次的用户 | ¥1,188（OK） | ¥950/买年包（划算） |
| 切入门槛 | 高，"要想清楚" | 极低，"才 19 块试试" |
| 低频用户占比 | 60%的中小企业年签合同<10份 | 按次完美覆盖 |

**定价锚点心理学：请律师看一份合同 ¥500-2000。19 块对冲几万块的合同风险，决策门槛为零。**

## 三、"一次审查"的精确定义

### 不是一次 API 调用，是一份合同的完整闭环

```
用户上传合同 → 审查 → 出报告 → 用户按建议修改 → 再审查 → ...
                                              │
                                  最多 3 轮，都只算 1 次额度
```

### 规则明细

| 项目 | 规则 |
|---|---|
| 范围 | 同一份合同（通过文档指纹匹配） |
| 轮次 | 最多 3 轮"审 → 改 → 再审" |
| 有效期 | 首次审查后 7 天内 |
| 第 4 轮起 | 每轮扣 1 次额外额度（¥9.5/轮） |
| 重新计费条件 | 上传全新合同、合同主体变更、合同类型变更 |
| 不计费条件 | 基于审查意见修改后重新上传、同一合同微调措辞 |

### 为什么是 3 轮不是无限

| | 无限轮 | 3 轮封顶 |
|---|---|---|
| 滥用风险 | 一份合同审 50 次，吃掉全部利润 | 安全边界 |
| 实际行为 | 几乎没人用超过 3 轮 | 99% 用户 2 轮内搞定 |
| 第 4 轮本质 | 不是审查而是推敲措辞 → 那是起草的活 | — |

### 用户视角

```
【第 1 次审查】扣 1 次额度
→ 发现 12 个风险点
→ 状态栏："您还有 2 次免费重审（7 天内）"

【第 2 次审查】不扣费（剩余免费次数：1）
→ 10 个已解决，2 个待调整

【第 3 次审查】不扣费（剩余免费次数：0）
→ 全部通过 ✅

→ 状态栏："本轮审查已用完全部免费额度。
   如需继续修改，每轮额外 ¥9.5"
```

## 四、API Key 管理

### 核心原则

**用户买的是审查次数，不是 API 额度。Key 是我们统一管理，用户碰不到。**

```
用户充值 → 审查余额 +N 次
     │
用户触发审查 → 系统扣余额 1 次
     │
系统用公司 Key 调 LLM API → 我们付 LLM 钱
     │
利润 = ¥19 - ¥2.35(API成本) = ¥16.65
```

### Key 管理架构

```
┌─────────────────────────────────────────────────────┐
│                  Key Vault（HashiCorp Vault）         │
│                                                      │
│  开发环境          测试环境           生产环境          │
│  ────────         ────────          ────────          │
│  gpt-4o-dev       gpt-4o-staging    gpt-4o-prod      │
│  claude-dev       claude-staging    claude-prod       │
│  deepseek-dev     deepseek-staging  deepseek-prod     │
│                                                      │
│  额度 $50/月      额度 $200/月      额度：按量后付       │
│  （个人卡）        （公司账户）        （公司账户）       │
│                                                      │
│  安全措施：                                           │
│  ├── 代码零硬编码（运行时从 Vault 动态读取）            │
│  ├── 多环境隔离（防止开发 Bug 刷爆生产额度）            │
│  ├── 余额监控（每 15 分钟检查，< $100 告警）           │
│  ├── 调用成功率监控（< 99% 告警）                      │
│  ├── 热轮换（更新 Vault → 代码零感知 → 自动生效）       │

```

## 五、成本风控

### 风险场景

按次收费有个隐藏风险：用户上传 200 页的采购合同，里面全是复杂表格，Token 消耗爆炸，单份吃掉 ¥15——毛利只剩 ¥4。

### 四层风控

| 层级 | 措施 | 触发条件 |
|---|---|---|
| **上传限制** | 超过 50MB / 100 页 → 提示"超大合同请选企业版" | 上传时 |
| **动态定价** | 50-100 页合同 → 扣 2 次额度（上传前明确提示用户） | 分析页面前 |
| **成本熔断** | 单次 LLM 消耗 > ¥15 → 自动切换低成本模型（DeepSeek V4-Flash）完成剩余任务 | 审查中途 |
| **利润监控** | 每笔审查记录实际 LLM 成本，日报显示毛利率 | 实时 |

### 成本追踪代码

```python
def start_review(user_id: str, contract_file: bytes) -> ReviewResult:
    # 1. 扣额度（原子操作，防并发超扣）
    user = db.atomic_deduct_credit(user_id, amount=1)
    if not user:
        raise InsufficientCredit("余额不足，请先购买审查次数")
    
    # 2. 用公司 Key 调 LLM
    result = llm_client.review(contract_file)  # Key 来自 Vault
    
    # 3. 记录实际成本
    actual_cost = result.token_usage.total_cost  # ¥2.41
    
    db.insert_cost_log(
        review_id=result.id,
        user_charged=1,        # 用户付了 1 次
        llm_cost=actual_cost,  # 实际花了 2.41
        profit_margin=(19 - actual_cost) / 19  # 87.3%
    )
    
    # 4. 成本异常告警
    if actual_cost > 15:
        alert(f"审查 {result.id} 成本 ¥{actual_cost}，接近亏损！")
    
    return result
```

## 八、面试 Q&A

**Q：为什么不是月订阅？**

> "市场调研显示 60% 的中小企业每年签不到 10 份合同。月订阅制对这一半以上的用户是负体验——他们会觉得'我一年用 2 次为什么要付 12 个月'。按次计费消灭了这个决策障碍。"

**Q：API Key 怎么管？**

> "SaaS 模式下 Key 是我们统一管理在 HashiCorp Vault 里，代码运行时动态读取，支持热轮换。开发/测试/生产三套 Key 隔离，防止开发 Bug 刷爆生产额度。私有化部署版支持客户自带 Key，前端配置页录入后 AES-256 加密存储。"

**Q：怎么防止用户一份合同反复审吃掉利润？**

> "3 轮免费重审内覆盖 99% 的修改场景。超过 3 轮按半价 ¥9.5 重新计费，既防止滥用，又不给正常用户添堵。"

**Q：成本怎么控制？**

> "四层风控：上传限制（50MB）、动态定价（超大合同扣 2 次）、成本熔断（单次 >¥15 自动切低成本模型）、利润监控（每笔记录实际成本，日报告警）。"
