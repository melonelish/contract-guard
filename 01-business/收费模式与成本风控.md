## I. Core Pricing Philosophy

**It's not a subscription — it's per-use review purchases. What users get for ¥19 is not a single API call, but a complete review loop for one contract (3 rounds of review-edit-re-review).**

### Why Per-Use Instead of Subscription

| | Monthly Subscription ¥99 | Per-Use Purchase ¥19/use |
|---|---|---|
| Users with only 2 uses/year | ¥1,188 (terrible deal) → churn | ¥38 (reasonable) → retained |
| Users with 50 uses/year | ¥1,188 (OK) | ¥950/buy annual pack (worth it) |
| Entry barrier | High, "need to think it through" | Extremely low, "just ¥19, give it a try" |
| Low-frequency user share | 60% of SMEs sign <10 contracts/year | Per-use perfectly covers |

**Pricing anchor psychology: Having a lawyer review a contract costs ¥500-2,000. ¥19 to hedge against tens of thousands in contract risk — zero decision barrier.**

## III. Precise Definition of "One Review"

### Not a Single API Call — It's a Complete Loop for One Contract

```
User uploads contract → Review → Report generated → User modifies per
                                                     suggestions → Re-
                                                     review → ...
                                                          │
                                   Up to 3 rounds, all counted as 1 use
```

### Rule Details

| Item | Rule |
|---|---|
| Scope | Same contract (matched via document fingerprint) |
| Rounds | Up to 3 rounds of "review → edit → re-review" |
| Validity Period | Within 7 days of first review |
| From Round 4 | Each round deducts 1 additional use (¥9.5/round) |
| Re-billing Conditions | Uploading a new contract, change of contracting party, change of contract type |
| Non-Billing Conditions | Re-upload after modifying based on review suggestions, minor wording tweaks to same contract |

### Why 3 Rounds Instead of Unlimited

| | Unlimited Rounds | 3-Round Cap |
|---|---|---|
| Abuse risk | One contract reviewed 50 times, consumes all profit | Safety boundary |
| Actual behavior | Almost no one uses more than 3 rounds | 99% of users finish within 2 rounds |
| Essence of Round 4 | Not review but wordsmithing — that's drafting work | — |

### User Perspective

```
[1st Review] 1 use deducted
→ Found 12 risks
→ Status: "You have 2 free re-reviews remaining (within 7 days)"

[2nd Review] No charge (remaining free uses: 1)
→ 10 resolved, 2 pending adjustment

[3rd Review] No charge (remaining free uses: 0)
→ All passed ✅

→ Status: "All free review uses for this review exhausted.
   Continuing modifications costs ¥9.5 per round"
```

## IV. API Key Management

### Core Principle

**Users buy review uses, not API quota. Keys are centrally managed by us — users never touch them.**

```
User top-up → Review balance +N uses
     │
User triggers review → System deducts 1 use
     │
System calls LLM API with company key → We pay LLM
     │
Profit = ¥19 - ¥2.35 (API cost) = ¥16.65
```

## V. Cost Risk Control

### Risk Scenario

Per-use pricing has a hidden risk: a user uploads a 200-page procurement contract full of complex tables, Token consumption explodes, a single use costs ¥15 — gross profit shrinks to ¥4.

### Four-Layer Risk Control

| Layer | Measure | Trigger Condition |
|---|---|---|
| **Upload limit** | Exceeds 50MB / 100 pages → prompt "For extra-large contracts please choose Enterprise edition" | On upload |
| **Dynamic pricing** | 50-100 page contract → deduct 2 uses (explicitly notify user before upload) | Before page analysis |
| **Cost circuit breaker** | Single LLM consumption > ¥15 → auto-switch to low-cost model (DeepSeek V4-Flash) to complete remaining tasks | Mid-review |
| **Profit monitoring** | Record actual LLM cost for each review, daily report shows gross margin | Real-time |

### Cost Tracking Code

```python
def start_review(user_id: str, contract_file: bytes) -> ReviewResult:
    # 1. Deduct credit (atomic operation, prevents concurrent over-deduction)
    user = db.atomic_deduct_credit(user_id, amount=1)
    if not user:
        raise InsufficientCredit("Insufficient balance, please purchase review uses first")
    
    # 2. Call LLM with company key
    result = llm_client.review(contract_file)  # Key from Vault
    
    # 3. Record actual cost
    actual_cost = result.token_usage.total_cost  # ¥2.41
    
    db.insert_cost_log(
        review_id=result.id,
        user_charged=1,        # User paid 1 use
        llm_cost=actual_cost,  # Actual cost ¥2.41
        profit_margin=(19 - actual_cost) / 19  # 87.3%
    )
    
    # 4. Abnormal cost alert
    if actual_cost > 15:
        alert(f"Review {result.id} cost ¥{actual_cost}, approaching loss!")
    
    return result
```

## VIII. Interview Q&A

**Q: Why not monthly subscription?**

> "Market research shows 60% of SMEs sign fewer than 10 contracts per year. For over half of users, a monthly subscription is a negative experience — they'll think 'why do I pay for 12 months when I only use it twice a year?' Per-use pricing eliminates this decision barrier."

**Q: How are API Keys managed?**

> "In SaaS mode, keys are centrally managed in HashiCorp Vault, dynamically read at runtime, with hot rotation support. Three sets of keys isolated for dev/test/prod to prevent dev bugs from draining production quota. Private deployment supports customer's own key, stored with AES-256 encryption after entry on the frontend config page."

**Q: How to prevent users from repeatedly reviewing the same contract and eating into profits?**

> "3 free re-reviews cover 99% of modification scenarios. Beyond 3 rounds, re-billing at half price ¥9.5 prevents abuse without frustrating normal users."

**Q: How are costs controlled?**

> "Four-layer risk control: upload limits (50MB), dynamic pricing (deduct 2 uses for extra-large contracts), cost circuit breaker (auto-switch to low-cost model when >¥15 per use), profit monitoring (record actual cost per use, daily alert reporting)."
