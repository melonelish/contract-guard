## 验证场景

### 场景 1：A 完成 → 进入 B（核心场景）

**目标**：验证从已完成审查的合同切换到另一个合同时，不会显示前一个合同的报告数据。

**步骤**：

1. 登录系统 → 进入工作台 `/workspace`
2. 找到一份**已完成审查**的合同 A，点击进入详情页 `/contracts/:id_A`
3. 确认页面显示：
   - ✅ 左侧：合同 A 的风险条款详情
   - ✅ 右侧：合同 A 的风险队列（N 个风险卡片）
   - ✅ 顶部状态："审查完成"
4. **打开浏览器开发者工具 → Console 标签**
5. 返回工作台 `/workspace`
6. 找到另一份合同 B（未审查 或 审查中），点击进入详情页 `/contracts/:id_B`
7. **在 Console 中观察**：应该看到类似 `[Review Store] clear() called` 的日志（如果有日志输出）

**预期结果**：

- ❌ **修复前**：合同 B 页面可能会短暂显示合同 A 的风险队列和条款
- ✅ **修复后**：
  - 合同 B 页面不显示任何报告数据（因为 B 未完成审查）
  - 或显示合同 B 自己的报告数据（如果 B 已完成）
  - **绝不显示**合同 A 的数据

**验证点**：

```javascript
// 在浏览器 Console 中检查 store 状态
// 进入合同 B 后立即执行：
localStorage.getItem('contractguard-review')
// 应该看到 "current": null 或 "current": {..., "contract_id": "B的ID"}
```

### 场景 2：刷新页面

**目标**：验证刷新页面后不会显示其他合同的残留数据。

**步骤**：

1. 在合同 A 详情页（已完成审查）
2. 复制浏览器地址栏的 URL（如 `http://localhost:5173/contracts/abc123`）
3. 打开新标签页，访问合同 B 的详情页 URL
4. 在合同 B 页面按 F5 刷新

**预期结果**：

- ✅ 刷新后只显示合同 B 的数据（或无数据）
- ✅ 不会显示合同 A 的风险队列

---

### 场景 3：A running → 进入 B → A 完成

**目标**：验证并发审查场景下的数据隔离。

**步骤**：

1. 上传合同 A，触发审查（状态：审查中）
2. 立即进入工作台，上传合同 B，进入合同 B 详情页
3. 等待合同 A 审查完成（后台）
4. 观察合同 B 页面是否显示合同 A 的报告

**预期结果**：

- ✅ 合同 B 页面不会突然显示合同 A 的报告
- ✅ 即使全局 store 收到合同 A 的报告，页面也会因为校验失败而过滤

---

### 场景 4：风险索引重置

**目标**：验证切换合同时风险索引不会越界。

**步骤**：

1. 进入合同 A 详情页（假设有 10 个风险）
2. 在右侧风险队列中，点击第 8 个风险卡片（activeRiskIndex = 7）
3. 确认左侧显示第 8 个风险的详细内容
4. 返回工作台，进入合同 B 详情页（假设只有 3 个风险）
5. 观察左侧显示的内容

**预期结果**：

- ❌ **修复前**：可能崩溃或显示空白（因为索引 7 越界）
- ✅ **修复后**：显示合同 B 的第 1 个风险（索引重置为 0）

## 技术验证点

### 检查 React DevTools

如果安装了 React DevTools：

1. 打开 DevTools → Components 标签
2. 找到 `ContractDetailPage` 组件
3. 观察 hooks 状态：
   - `reportData`：全局 store 的原始数据
   - `report`：经过校验后的数据
   - `activeRiskIndex`：当前选中的风险索引

**验证点**：
- 切换合同时，`report` 应该变为 `null`（即使 `reportData` 还有值）
- `activeRiskIndex` 应该重置为 `0`

### 检查 Network 面板

1. 打开 DevTools → Network 标签
2. 切换合同时观察：
   - 应该看到 `GET /api/v1/contracts/:id` 请求
   - **不应该**看到对旧合同的 report 请求残留

## 修复机制说明

### ContractDetailPage 三重防护

1. **入口清理**（第 798-803 行）
   ```typescript
   useEffect(() => {
     if (id) {
       clearReviewStore();  // 清空全局 store
       void fetchContractDetail(id);
     }
   }, [id, clearReviewStore, fetchContractDetail]);
   ```

2. **出口校验**（第 814-820 行）
   ```typescript
   const report = useMemo(() => {
     if (!reportData) return null;
     if (activeReview && reportData.id !== activeReview.id) return null;
     if (selected && reportData.contract_id !== selected.id) return null;
     return reportData as ReviewReport;
   }, [reportData, activeReview, selected]);
   ```

3. **状态重置**（第 829-832 行）
   ```typescript
   useEffect(() => {
     setActiveRiskIndex(0);
   }, [report]);
   ```

### ReviewReportPage 双重防护

1. **入口清理**（第 464-469 行）
   ```typescript
   useEffect(() => {
     if (id) {
       clear(); // Clear first to avoid showing old report
       void fetchReport(id);
     }
   }, [fetchReport, clear, id]);
   ```

2. **出口校验**（第 472-478 行）
   ```typescript
   const report = useMemo(() => {
     if (!reportData) return null;
     // Only use report if it matches the current review ID from URL
     if (reportData.id !== id) return null;
     return reportData as ReviewReport;
   }, [reportData, id]);
   ```

## 已知限制

1. **WebSocket 连接**：快速切换时可能有短暂的旧连接残留，但不会影响显示（因为有校验）
2. **轮询请求**：切换时可能有一次旧请求返回，但会被校验过滤

这些是低风险边界，不影响核心修复效果。
