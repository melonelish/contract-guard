### 任务 2：修正"保存更改"产品语义 ✅

**问题诊断：**
- 当前保存只是 `setState`，退出页面会丢失

- 使用 `sessionStorage` 按 `contractId` 隔离保存草稿
- 草稿键：`contract-draft-${contractId}`
- 按钮文案改为"保存草稿"
- 成功提示明确说明"草稿已保存到本次会话（退出浏览器后将丢失）"
- 未保存提示改为"您有未保存的草稿更改（仅保存到本次会话）"
- 新增蓝色提示框，明确说明当前编辑模式的限制

**修改文件：**
- `D:\FaLvXM\frontend\src\components\Editor\EditMode.tsx`
- `D:\FaLvXM\frontend\src\pages\ContractDetailPage.tsx`

**实现细节：**
```typescript
// 进入编辑模式时，尝试加载本地草稿
const draftKey = `contract-draft-${id}`;
const savedDraft = sessionStorage.getItem(draftKey);

// 保存时写入 sessionStorage
const handleSaveContent = (content: string) => {

  const draftKey = `contract-draft-${id}`;

  setEditedContent(content);
  messageApi.success('草稿已保存到本次会话（退出浏览器后将丢失）');
```

### 任务 4：修正"应用建议"能力定义 ✅

**问题诊断：**
- 当前只是对 HTML 字符串做 `replace`，属于原型级

**修正方案：**

## 风险提示

### 哪些能力仍然只是占位
1. **"重新审查"功能**：当前完全未实现，按钮已禁用并明确标注"开发中"
2. **合同全文编辑**：当前没有接口返回合同全文，只能编辑风险条款片段的拼接
3. **服务端草稿持久化**：当前只有 `sessionStorage` 本地缓存，没有后端接口支持

### 真正完成编辑闭环还缺什么

#### 后端需要补充的接口
1. **GET /contracts/{id}/full-text**
   - 返回合同完整原文（纯文本或结构化 JSON）
   - 包含所有条款，不只是有风险的条款

2. **POST /contracts/{id}/drafts**
   - 保存用户编辑后的草稿到后端
   - 支持多版本草稿管理

3. **POST /contracts/{id}/re-review**
   - 接受用户编辑后的内容（或草稿 ID）
   - 重新发起审查，生成新的 `review_id`

#### 前端需要补充的能力
1. **真正的合同全文查看器**
   - 当接口返回全文后，用结构化方式展示
   - 支持按条款跳转

2. **真正的条款定位联动**
   - 点击风险卡片时，高亮合同原文中的对应条款
   - 反之亦然

3. **编辑后重新审查的完整流程**
   - 保存草稿 → 调用重新审查接口 → 监听新审查进度 → 展示新报告

## 总结

本轮修正的核心原则：**诚实降级，准确命名，明确边界**。

### 不再做的错误表述
- ❌ "编辑合同正文"（根本没有合同正文）
- ❌ "保存成功"（只是 `sessionStorage`）
- ❌ "重新审查"（后端接口不存在）
- ❌ "应用到合同"（没有合同可应用）

### 当前准确的能力定义
- ✅ "审查建议整理"（编辑的是风险条款片段拼接）
- ✅ "保存草稿到本次会话"（`sessionStorage`，退出浏览器后丢失）
- ✅ "功能暂未支持"（明确标识占位能力）
- ✅ "插入建议内容"（而不是"应用到合同"）

### 当前可继续演进的路径
1. 后端补充合同全文接口 → 前端真正实现"合同全文编辑"
2. 后端补充草稿持久化接口 → 前端升级为真正的"保存"
3. 后端补充重新审查接口 → 前端打通"编辑-审查"闭环
