## 1. Feature Positioning

**Core Goal**: Let users see at a glance "where the AI wants me to change", instead of reading the review report and then manually locating positions to edit.

**Value Proposition**:
- Reduce user comprehension cost (no mental mapping of modification effects needed)
- Improve editing efficiency (one-click accept / batch processing)
- Enhance trust (WYSIWYG — you can see where changes are made)

```yaml
- text: "specific text content"

inline_diff should be:

  {"type": "unchanged", "text": "Acceptance time is "},

  {"type": "added", "text": "3"},

3. If an entire sentence is added or deleted, represent it as a single diff item
```

## 7. Built-in Editor

### 7.1 Why a Built-in Editor Is Necessary

```
Pain points of external editing:
───────────────
User downloads Word → opens in WPS → tables misaligned → paragraphs broken →
uploads again for review → system doesn't recognize this as a revised version → loses review context

Value of built-in editor:
───────────────
User sees review suggestions → modifies directly in the system → formatting never breaks →
clicks "Re-Review" after editing → system knows exactly what changed → auto-compares differences →
user stays in the system, never leaving for Word/external tools
```

### 7.3 AI Annotations Embedded in the Editor

Core design: AI risk markers are embedded in the editor content area as `<mark>` tags that cannot be deleted by the user.

```html

  <div class="clause-content" contenteditable="true">
    After contract signing, Party A shall pay Party B
    <mark class="risk-high" 
          data-risk-id="R-001"
          contenteditable="false">30%</mark>
    of the total contract price as advance payment.
  </div>
  
  <!-- AI suggestion banner (expands on marker click) -->
  <div class="ai-suggestion-banner">
    <div class="suggestion-header">
      🔴 High Risk — Damages ratio may be too high
    </div>

      Suggest changing to <strong>20%</strong>, not exceeding actual losses
    </div>
    <div class="suggestion-actions">
      <button class="btn-accept"
```

**Rule**: `<mark contenteditable="false">` ensures users editing surrounding text cannot accidentally delete AI annotations. Markers can only be removed via "Apply Suggestion" or "Ignore".

### 7.6 Change Tracking and Incremental Review

```python
# After user edits and clicks "Re-Review", the system knows exactly what changed
class ContractRevision:
    """Each save generates a snapshot"""
    
    def save_snapshot(self, contract_id: str, content: dict):
        previous = self.get_latest_snapshot(contract_id)
        
        # Compare previous and current versions
        changed_clauses = diff(
            old=previous.clauses,
            new=content.clauses
        )
        
        snapshot = {
            "version": previous.version + 1,
            "content": content,
            "changed_clauses": changed_clauses,  # Only 3 clauses changed
            "timestamp": now(),
        }
        
        return snapshot
    
    def incremental_review(self, contract_id: str):
        """Incremental review: only review modified clauses"""
        latest = self.get_latest_snapshot(contract_id)
        changed = latest.changed_clauses  # Only 3 clauses
        
        # Re-analyze only the changed clauses
        results = []
        for clause_id in changed:
            clause = latest.content.clauses[clause_id]
            analysis = analyzer_agent.analyze(clause)
            results.append(analysis)
        
        # Unchanged clauses retain previous review results
        previous_review = self.get_review(contract_id)
        merged = merge_results(previous_review, results)
        
        return merged
```

**Value of Incremental Review**:
- User modifies 3 clauses; the system only re-reviews those 3, not the entire contract
- Speed drops from 30 seconds to 5 seconds
- Token cost drops from ¥2.35 to ¥0.3
